# 自审：架构与产品缺陷清单（2026-07-08）

> 对照 [mvp-blueprint](mvp-blueprint.md) 逐层审计代码与体验后的**如实结论**。
> 分级：**P0**=违背定稿/损害信任，立即修；**P1**=近期必补；**P2**=记录在案。
> P0 已随本次提交修复（标 ✅）。

## P0 · 立即修复

### 1. 记忆信源泄漏进界面 ✅（违背蓝图）
**现象**：与助手对话后，`remember_exchange` 创建的「与 X 的对话」conversation 信源会出现在**信源列表、绑定选择器、全局搜索**里——而蓝图明确「记忆作为助手的能力呈现，不暴露实现」。用户越聊，信源页越乱。
**修复**：`list_sources` 默认排除 `source_type=conversation`（`include_memory` 参数保留给内部）；全局搜索与绑定选择随之干净。助手自身检索不受影响（`resolve_sources` 直查）。

### 2. 删除信源不清理 ✅
**现象**：删除信源只删了元数据行。三处残留：
- **引擎槽常驻**：`EngineManager` 缓存的 `DataEngine` 不关闭、不移除；
- **上传文件残留**：`uploads/{source_id}/` 整目录留在磁盘；
- **悬挂绑定**：`soul_bindings.target_id` 是普通字符串（无 FK），指向已删信源的绑定永久残留，绑定弹窗里显示成幽灵「信源」。
**修复**：删除时 ①`EngineManager.release()` 关闭并移除槽位 ②删除该源上传目录 ③清理指向它的 source 级绑定；确认弹窗文案说明影响。引擎侧向量/图谱数据的彻底清除仍留待企业档（合规删除）。

### 3. 黄金路径断链：助手创建后零绑定 ✅
**现象**：蓝图第二步是「创建助手，**绑定信源**」，但创建弹窗只有名字/设定——新用户建完助手直接提问，检索为空，答案永远是「资料中未提及」。第一次体验即挫败。
**修复**：①创建弹窗内置**信源勾选**（可多选，创建即绑定）；②工作台在零绑定时显示醒目引导条（「尚未绑定信源——回答将没有依据」+ 绑定按钮）。

### 4. 生产部署防呆缺失 ✅
**现象**：`environment=prod` 时仍可用默认 `secret_key` 启动（JWT 可被伪造）；注册永远开放（公网部署=任何人可注册进你的实例）。与「企业可直接部署、严谨」相悖。
**修复**：①prod + 默认密钥 → **拒绝启动**并给出明确指引；②新增 `SAG_ALLOW_REGISTRATION`（默认开），关闭后仅允许**首个用户**注册（部署引导），其余返回 403。

## P2 · 记录在案

- ~~前端 fetch 无超时~~ **✅ 已修（R3）**：`request()` 30s AbortSignal 超时护栏（SSE 除外）。
- ~~上传仅校验大小，未白名单 MIME~~ **✅ 已修（R3）**：扩展名白名单（后端校验 + capabilities 暴露 + 前端 accept 拦截）。

- 同一会话并发提问的记忆写入顺序未加锁（低风险）。
- 遗留 `chat_threads/chat_messages` 表与模型保留（老库兼容），已无 API 面。
- 完整 user/run 维记忆与 key-as-scope 密钥体系：见 [product-form](product-form.md)，v0.1.0 预留。

## 结论

地基（引擎适配、任务、生成、SSE、令牌化设计系统）是稳的；本轮暴露的问题集中在**生命周期收尾**（删除清理、缓存边界）与**首次体验闭环**（零绑定、溯源）。P0 修完后，产品在「30 秒能懂、三步能跑通」之外，补上了「删得干净、部署得安全、第一问就有依据」。
