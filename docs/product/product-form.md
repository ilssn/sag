# zleap 产品形态定稿（v0.1.0）

> 本篇是收敛后的**最终形态定义**。取代早期发散的「灵魂 / 挂载 / 夺舍」等表述——
> 那些作为长期可能性保留在 [architecture/](../architecture/overview.md)，但**不属于当前产品形态**。

## 一句话

zleap 是一个**上下文管理平台**：把文档、对话、记忆汇入一个**记忆体**，再以搜索、助手、
OpenAI 端点对外提供。个人与公司是同一套模型的两种规模。

## 三层模型：供给 → 记忆体 → 出口

```
        供给（Ingress）                记忆体（Memory Body）              出口（Egress）
   ┌────────────────────┐        ┌──────────────────────────┐      ┌────────────────────┐
   │ 上传文档            │        │  固定结构：                │      │ 搜索（⌘K，可@信源）│
   │ 统一写入 /ingest    │  ───▶  │   分块 chunk               │ ───▶ │ 助手（带引用对话）  │
   │ 对话自动沉淀        │        │   事件 event               │      │ OpenAI 兼容端点     │
   │ （连接器：预留）    │        │   实体 entity              │      │ （MCP：预留）       │
   └────────────────────┘        │  记忆维度：agent（现）      │      └────────────────────┘
                                  │   user / run（预留，mem0）  │
                                  └──────────────────────────┘
```

### 1. 供给（Ingress）——内容怎么进来

- **上传**：文档进入信源，引擎自动解析、分块、向量化、抽取事件与实体。
- **统一写入**：`POST /sources/{id}/documents/ingest` 接收文本 / 消息，供外部系统持续推送。
- **对话沉淀**：每轮助手问答自动写回该助手的记忆信源（见记忆维度）。
- **连接器**：采集层已抽象为注册表（file_upload 在用，web 隐藏）；更多连接器是拓展点，非当前形态。

### 2. 记忆体（Memory Body）——内容如何被结构化与演进

记忆体的**结构是固定的**（由 zleap-sag 保证）：`分块 / 事件 / 实体`。这是 zleap 的确定性地基——
不追求让用户理解或配置它，而是把它作为「本能」运转。

**记忆维度**（借鉴 mem0 的 user / agent / run 切片，见 [mem0-insights](mem0-insights.md)）：

- **agent 维（现已实现）**：记忆挂在助手上。私有助手 = 个人记忆；团队助手 = 团队共享记忆。
- **user 维 / run 维（预留）**：按人、按会话切片记忆，用于更细的个性化与临时上下文。当前用
  「会话按人隔离」近似覆盖团队场景的隐私诉求。

### 3. 出口（Egress）——记忆体怎么被消费

- **搜索**：全局 ⌘K，可 `@信源` 锁定范围，命中即跳原文（chunk 级溯源）。
- **助手**：绑定信源 + 人格，带引用流式回答；防幻觉短路、prompt 透明、记忆回灌。
- **OpenAI 兼容端点**：任意助手 = 一个「带记忆与引用的模型」，供外部系统无缝接入。
- **MCP（预留）**：把记忆体作为工具暴露给本地 Agent，是明确的拓展方向，非当前形态。

## 个人 ↔ 公司：同一模型的两种规模

**空间（Workspace）就是一个记忆体。** 不为个人和企业分别造两套东西：

| | 个人 | 公司 |
| --- | --- | --- |
| 空间 | 1 人空间 | N 人空间 |
| 成员 | 只有自己（所有者） | 所有者 + 编辑者 + 只读 |
| 助手可见性 | 都是自己的 | 私有 / 团队共享 |
| 记忆 | 个人记忆 | 团队共享 + 会话按人隔离 |
| 审计 | 也在，只是只有你 | 所有者查看 / 导出留档 |

个人用户对「团队」无感（可见性开关仅在多人空间出现）；公司用户无需迁移到「另一套企业版」。

## 团队机制规格

- **角色**：`OWNER`（管理成员/空间/审计）· `EDITOR`（读写内容）· `VIEWER`（仅检索与对话）。
- **切换**：请求头 `X-Workspace-Id` 指定活动空间；顶栏切换器（多空间成员才出现）。
- **邀请**：按邮箱邀请**已注册**用户（MVP 不发邮件）；最后一名所有者不可自降/被移除。
- **可见性**：助手 `private | workspace`；仅创建者或空间所有者可管理（改设定/绑定/删除/可见性）。
- **共享记忆**：团队助手的对话沉淀为团队记忆（agent 维语义）；但**会话（thread）按人隔离**——
  彼此看不到对方的对话与消息。

## 下一步原则：key-as-scope（预留，不在 v0.1.0）

对外访问应以「密钥即作用域」建模：一把 API key 绑定到某空间 + 某助手 + 某角色，
调用者拿到的能力由 key 决定，而非共享用户 JWT。这是 OpenAI 端点从「JWT bearer」走向
「可分发密钥」的自然演进，也是 mem0 的 key-as-scope 思路。当前 v0.1.0 用用户 JWT 鉴权，
足够个人与小团队；密钥体系留待下一阶段。

## 明确不做（当前形态的边界）

- 不做 OCR / 版式解析 / 模板市场的军备竞赛（交给引擎与上游）。
- 不做工作流画布 / 渠道矩阵（IM/公众号接入）。
- 不做「灵魂挂载 / 夺舍」这类隐喻化交互——统一为「助手 + 记忆」。

保持四个概念、轻量依赖、稳定可审计——这就是超越 RAGFlow 的方式。
