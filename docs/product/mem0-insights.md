# mem0 调研启发（团队与记忆模型）

> 调研 [mem0](https://mem0.ai) 的组织/项目模型与记忆维度，沉淀对 zleap 团队与记忆设计的启发。
> 结论已落地于 [R1 团队与权限](../architecture.md) 与 [产品形态定稿](product-form.md)。

## mem0 的关键设计

1. **组织 → 项目 两层**。成员挂在组织，资源与配额按项目隔离。
2. **两档角色语义**（READER / OWNER 为主）。轻，不做复杂 RBAC 矩阵。
3. **记忆按维度切片**：`user_id` / `agent_id` / `run_id` 三个维度可组合——
   同一条记忆可归属某个用户、某个 agent、某次运行，检索时按维度过滤。
4. **key-as-scope**：API key 即作用域，一把 key 决定能访问哪个项目/维度，而非共享账号令牌。

## 对 zleap 的启发与取舍

| mem0 概念 | zleap 的采纳 | 取舍理由 |
| --- | --- | --- |
| 组织 → 项目两层 | **压平为单层「空间」** | 个人=1 人空间、公司=N 人空间，一层足够；两层对个人是负担 |
| READER / OWNER 两档 | **三档 OWNER/EDITOR/VIEWER** | 知识库需要「能写但不能管人」的编辑者档位 |
| agent_id 维记忆 | **已实现**：记忆挂在助手上 | 团队共享助手 = 团队共享记忆，语义正好对上 |
| user_id 维记忆 | **用「会话按人隔离」近似** | v0.1.0 先满足隐私诉求；完整 user 维记忆留待后续 |
| run_id 维记忆 | **预留** | 临时/一次性上下文，当前非核心场景 |
| key-as-scope | **预留原则**（见 product-form） | v0.1.0 用用户 JWT；密钥体系是 OpenAI 端点的下一步 |

## 落地映射

- **空间 ≡ mem0 项目**：资源隔离、成员归属、审计边界都以空间为单位。
- **助手 ≡ mem0 agent 维**：这是「共享助手的记忆是团队记忆」在语义上成立的依据。
- **共享助手会话按人隔离**：用 thread 的 `user_id` 过滤，近似 user 维的隐私切片，
  在不引入完整 user 维记忆的前提下满足团队隐私诉求。
- **下一步 key-as-scope**：把 OpenAI 端点的鉴权从共享 JWT 升级为可分发的作用域密钥。

## 一句话结论

mem0 教我们的不是「照抄组织/项目」，而是**用维度而非层级来建模记忆与权限**——
zleap 把维度压到「空间 + 助手（agent 维）」两个够用的轴上，保持四概念的清爽，
同时为 user / run 维与 key-as-scope 留好演进路径。
