# 使用场景

用真实故事校准产品。每个场景都标注了它依赖的能力，方便排期。

## 1. 个人 · 第二大脑

小艾把飞书群、Notion 空间、几个常看的网页、每周会议录音都接进 sag。她创建了一个灵魂「阿默」，绑定「知识」+「会话记忆」两个命名空间。

- 平时直接和阿默聊：「上周和张三聊的排期是什么来着？」阿默跨飞书消息 + 会议录音 + 过往对话检索，带引用回答。
- 她越聊，阿默越懂她 —— 因为每次对话都沉淀进会话记忆。

**依赖**：消息/录音连接器（流式写入）· 会话记忆 · 多信源 fan-out · 灵魂。

## 2. 企业 · 决策大脑

某公司把制度库、合同、CRM 导出、会议纪要接入 sag，按部门建命名空间。管理员创建灵魂「决策脑」，只绑定「制度」「合同」两个命名空间（权限隔离）。

- 员工问：「A 客户的合同里，违约金条款是怎么约定的？」→ 带引用、可溯源到具体文档段落。
- 所有问答、绑定变更、数据接入都进**审计日志**，可导出合规。

**依赖**：RBAC + 多租户 · 审计 · 命名空间级绑定 · 连接器 · 企业部署（Postgres/pgvector）。

## 3. Agent · 灵魂（挂载到本地）

老王有一个本地 coding agent。他在 sag 下载「挂载插件」，填入 key，选择**知识挂载**：本地 agent 通过 MCP 拿到老王的 sag 上下文，回答问题时能引用他的私有知识。

进一步，他选择**全量挂载（夺舍）**：sag 备份本地 agent 的旧人格文件，写入灵魂「阿默」的人格与名字。重启后，本地 agent 开口就是熟悉的阿默，并提示：「阿默已登录到本地 🎉」。随时可一键**还原**旧人格。

**依赖**：MCP/Skill/Hook 上下文插件 · 挂载包与 Host 适配 · 人格备份/还原（可逆、安全）。

## 4. 想象力 · 一本书变成一群人

小林上传《三国演义》。sag 解析后，在**洞察**页按实体类型分类：人物、地点、事件……人物按「热度/重要度」排行（出现频次 + 图谱中心度）。

- 他点「关羽」→ 一键**提取成灵魂**：sag 依据书中关于关羽的事件，自动生成名字与人格草稿。
- 他和关羽聊：「你怎么看走麦城？」关羽以书中语气、依据书中情节回答。也能和「诸葛亮」聊结局。

**依赖**：事件—实体图谱的实体聚合与排序 · 「实体→灵魂」的人格自动生成 · 洞察页。

## 5. 持续接入 · 标准写入

第三方系统想把自己的消息流写进 sag。它不需要懂 sag 内部，只需按**统一消息格式**往 `POST /v1/sources/{id}/messages` 持续推：

```json
{ "messages": [
  { "author": "张三", "role": "user", "text": "明天评审几点？", "ts": "2026-07-07T09:00:00Z",
    "thread": "T-102", "meta": { "channel": "研发群" } }
]}
```

sag 负责缓冲、成块、入库、抽取。这就是「动态持续写入」——上下文源源不断地长。

**依赖**：统一 Ingestion API · 消息信源类型 · 缓冲/成块策略。
