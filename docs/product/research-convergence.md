# 调研与收敛（2026-07）

> 背景：产品概念开始飘（「灵魂」「上下文」「命名空间」三个抽象词叠加），需要停下来对照现实收敛。
> 本篇是三方调研的事实与**已定结论**——此后所有命名、文案、界面以此为准。

## 一、调研事实

### 1. zleap（引擎方，zleap.ai / zleap.com）

- 定位：**「企业的私有化 AI 大脑」**；zleap.com 一句话是「下一代内容社区——人与 Agent 共同创作和交流」。
- 概念用词：**「AI Agent / 智能同事」**（强调主动性）、「结构化知识」、信息以 **Feed 流（帖子）** 呈现；技术叫 **SAG™ 架构**。
- 三支柱：Information Manager（自动连接飞书/代码库/文档/会议）→ Company News（X 式信息流）→ Smart Colleague（Agent 主动发布洞察）。
- 关键承诺：**「所有内容 100% 可溯源到原始数据」**、SQL 级精准、数据不出门（内网部署）。

**启示**：引擎方自己都用 Agent/智能同事这种人人秒懂的词；溯源是全家族的立身之本——我们的引用要做到同等醒目。

### 2. RAGFlow（功能竞品）

- 概念体系（用户操作对象）：**Knowledge base（知识库）/ Dataset / Agent / Chat** ——全部是行业直白词。
- 功能强项：深度文档理解（Word/PPT/Excel/扫描件/图片）、**模板化分块 + 分块可视化人工干预**、多路召回 + 融合重排、**grounded citations（可溯源引用）**、agentic workflow + MCP、代码执行器、连接器（Confluence/S3/Notion/Google Drive/Discord）、多渠道（飞书/Discord/Telegram）；v0.26 新增 **Agent Memory**。
- 卖点：**"Quality in, quality out"**；开源 + 云服务（$0/29/129/企业）。
- 使用流程：部署 → 建知识库导入 → 分块可视化微调 → 配 LLM → 建 Chat/Agent → 带引用问答。

**启示**：a) 它的概念极其朴素，学习成本低；b) 它强在**解析与检索的工程深度**（分块可视化、重排、格式覆盖）；c) 它的 Agent/workflow 走「编排平台」路线（重、面向开发者）；d) 连 RAGFlow 都在补 Memory——记忆是行业公认方向，而这恰是我们借 SAG 的天然强项。

### 3. Codex（设计标杆）

来自 OpenAI 官方 frontend 设计规范与产品评测（[open-design.ai](https://open-design.ai/agents/codex-design/)、[developers.openai.com](https://developers.openai.com/blog/designing-delightful-frontends-with-gpt-5-4)、[Medium 评测](https://medium.com/@ariaxhan/i-tested-openais-new-codex-desktop-app-the-ui-is-the-real-product-c2c59bdcb5f6)）：

- **要**：calm surface hierarchy（冷静的表面层级）、强排版与留白、少色彩、**信息密集但可读**、minimal chrome、**卡片只在「卡片即交互」时使用**；最多两个字体族 + 一个强调色；动效克制且有目的。
- **避**：dashboard 卡片马赛克、处处粗边框、装饰性渐变、多个强调色互相打架、**不提升扫读效率的装饰性图标**。

**启示**：我们的令牌体系（纸墨+单一淡金+双字体+细线）已对齐；偏差在**卡片略多、徽标/图标略密**——收敛方向是列表化、提密度、砍装饰。

## 二、结论（已定，立即执行）

### 结论 1 · 术语收敛：向行业标准词回归

| 旧（对外） | **新（对外，唯一）** | 内部代码 | 理由 |
|---|---|---|---|
| 灵魂 Soul | **Agent** | `Soul`（保留，不迁移） | zleap/RAGFlow/用户全用 Agent；「灵魂」抬高理解成本 |
| 上下文 / 信源 Source | **知识库** | `Source` | 一个信源=RAGFlow 的一个 knowledge base，人人秒懂 |
| 命名空间 Namespace | **分组** | `Namespace` | 「命名空间」是工程词；分组=文件夹心智，零学习 |
| 人格 Persona | **设定** | `persona` | 按钮/表单用「设定」，字段直白（角色说明/开场白/回答规则） |
| 会话记忆 | **记忆**（Agent 属性） | memory namespace | 对用户是「这个 Agent 记得的东西」，不暴露实现 |
| 提取成灵魂 | **创建 Agent** | — | 动作即结果 |

- 「上下文 / Context Infra / 灵魂」降级为**叙事词**：只出现在 vision/README 的愿景段落，**绝不出现在界面**。
- 界面导航定稿：**概览 / Agent / 知识库 / 设置**。
- 后端标识符、表名、API 路径不迁移（`/souls` 等保留）——对外 OpenAPI 摘要标注即可；避免无意义的破坏性改动。

### 结论 2 · 一条黄金路径，先能用再有趣

新用户旅程收敛为一句话：**「建知识库 → 传文档 → 创建 Agent 并绑定 → 带引用对话」**。

- 概览页职责=引导完成这条路径（未完成时显示进度，完成后才展示统计与列表）。
- 「书→人物」等想象力功能是路径完成后的**彩蛋**，不参与首次引导。

### 结论 3 · 与 RAGFlow 的攻守选择

- **不追**：可视化 workflow 编排、多渠道机器人矩阵（他们的红海，面向开发者的重资产）。
- **守住差异**：SAG 事件-实体图谱、**Agent 记忆闭环**（他们刚起步）、零依赖极简部署、克制审美。
- **必须补齐的桌面筹码**（按优先级）：① 引用/分块的可视化与溯源体验（对齐「100% 可溯源」）→ ② 文档格式解析广度 → ③ 连接器矩阵（Notion/飞书/S3）。

### 结论 4 · 设计收敛（codex 校准）

1. 卡片只在「卡片即交互」时用：概览统计改为**一行数字**（非四张卡）；列表优先于卡片网格。
2. 每屏装饰性图标 ≤ 导航处；类型徽标只保留必要的（知识库类型、记忆标识）。
3. 信息密度上调：列表行高收紧、正文 65 字宽、数字 tabular-nums（已做）。
4. 动效保持两处（流式光标、在场点），不再新增。
5. 文案规则：按钮=动作本身；错误=原因+怎么办；不用比喻词。
