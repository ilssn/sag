# 核心概念模型

> 这是全项目的「世界观」。做原型前务必先对齐这张图。
>
> **术语 v2**（2026-07 [调研收敛](research-convergence.md)）：界面用词已改为 **Agent / 知识库 / 分组 / 设定**；本篇的 Soul/Source/Namespace 为内部模型名，关系与含义不变。

## 关系全景

```

   Workspace (工作空间)  │  多租户边界：成员、配置、审计、配额             │
   └──────────────────────────────────┬───────────────────────────────┘
                                       │ 1:N
             ┌─────────────────────────┼─────────────────────────┐
             ▼                         ▼                         ▼
     Namespace「会话记忆」       Namespace「知识」          Namespace「XXX」   ← 文件夹式分组
             │ 1:N                      │ 1:N
        ┌────┴────┐               ┌─────┴─────┐
        ▼         ▼               ▼           ▼
   Source(会话)  Source(会话)   Source(文档)  Source(网页)   ← 一条上下文流；1:1 引擎 source_config_id
        │                            │

        │   chunks(块) + 向量        │   由 Connector 持续写入
        │   + Event–Entity 图谱      │   由 SAG ingest+extract 产生

   ┌───────────────────────────────────────────────┐

             ┌───────────┴───────────┐

        Soul「第二大脑」          Soul「关羽」        ← 灵魂：名字 + 人格 + 绑定上下文 + 记忆

```

## 定义

### Context（上下文）
产品的基础与总称：所有被结构化、可检索的信息。它不是一张表，而是 Source 集合对外呈现的能力面。

### Source（信源）
一条独立的上下文流或集合，**1:1 对应引擎的一个 `source_config_id`**。关键升级：**信源有类型**。

| source_type | 来源 | 说明 |
|---|---|---|
| `document` | 上传 / 网页 | 静态或抓取的文档（现已支持） |
| `message` | IM / 飞书 / 自定义 | 结构化消息流，持续追加写入 |
| `conversation` | 与灵魂对话 | 会话记忆，自动创建、自动抽取 |
| `audio` | 录音 / 会议 | 转写后入库（转写→document 化） |
| `web` | 爬虫 | 动态抓取（现已支持） |

> 无论类型，最终都归一到引擎的 `ingest`（喂文件/文本）+ `extract`（事件图谱）+ `search`。类型只影响**采集与切分策略**，不改下游。

### Namespace（命名空间 / 集）
信源的**文件夹式分组**，隶属工作空间。用于组织与批量绑定。

- 每个工作空间默认有两个：**「会话记忆」**（放灵魂对话沉淀）、**「知识」**（放文档/网页）。
- 可自定义任意命名空间（如「合同」「会议」「客户 A」）。
- 例：多个人和同一个灵魂聊天 = 多个 `conversation` 信源，但都落在该灵魂的「会话记忆」命名空间里。

### Soul（灵魂）
可对话的实体 = **名字 + 人格(Persona) + 绑定的上下文 + 自己的记忆**。

- 用户创建灵魂、起名、设人格，然后**绑定命名空间 / 信源**作为其知识来源。
- 与灵魂对话时，muse 在检索层做 **fan-out**：跨其绑定的所有上下文 + 它自己的会话记忆检索，再生成带引用的回答。
- 「灵魂」只做对话，不做操作面板 —— 这是产品的情感锚点。

### Persona（人格）
灵魂的身份与行为配置（见 [souls-and-memory](../architecture/souls-and-memory.md)）：`system_prompt`、语气/voice、性格 traits、头像、边界/guardrails、默认检索策略。

### Connector（连接器）
把外部信息源**持续写入**某个信源的适配器。三态：
- **静态**：文件上传。
- **动态（pull）**：Web / Notion / S3 —— 周期 `discover`+`fetch`。
- **流式（push）**：飞书 / IM / 录音 —— 经统一「写入接口」持续追加。

见 [connectors](../architecture/connectors.md)。

### Memory（记忆）
**会话即记忆**：与灵魂的每次对话，都作为 `conversation` 信源沉淀进「会话记忆」命名空间，并自动 `extract` 成事件—实体，进入可检索上下文，形成「越聊越懂你」的闭环。

## 与现有代码的映射（做原型时的落点）

| 概念 | 现状（已实现） | 需新增 |
|---|---|---|
| Source | `Source` 模型 + `sag_source_config_id` ✓ | `source_type`、`namespace_id`、`origin` 元数据 |
| Namespace | —— | **新增** `Namespace` 模型 + 默认命名空间 |
| Soul | —— | **新增** `Soul` + `Persona(json)` + `SoulBinding` |
| Connector | `Connector` 抽象 + 注册表 + file_upload/web ✓ | 流式写入接口、message/audio 连接器 |
| Memory | `ChatThread/ChatMessage` ✓ | 会话 → `conversation` 信源的落地 + 抽取 |
| 检索 | 单信源 `search` ✓ | **多信源 fan-out** + 合并/重排 |
| 生成 | SSE 流式 + 引用 ✓ | 绑定人格上下文 |

> 结论：地基（信源、连接器、任务、生成、SSE）已就位。演进主线是**加两层组织（Namespace / Soul）+ 让信源有类型 + 多源检索**。数据不迁移、引擎不改，纯增量。
