# 产品设计规范（知识库 · Agent · MCP）

## 知识库设计规范

- **信源（Source）是唯一容器**：一个信源 = 引擎里一个隔离的 source_config（1:1），
  解析/分块/向量/事件图谱全部按源隔离；删源必须级联清理（文档、绑定、引擎槽、上传目录）。
- **入库即可用**：上传自动走 `ingest → extract` 状态机（pending→loading→extracting→ready|failed），
  失败可重处理；未配模型时上传/解析照常，仅抽取与问答受限——**永远给出降级路径**。
- **原文是第一公民**：任何呈现（搜索命中、对话引用、文档卡片）都必须能一步到达原文
  （chunk 级溯源 + 原始文件预览/下载）。展示派生内容而不给原文入口，视为设计缺陷。
- 展示密度可选（卡片/列表双视图），选择持久化；计数（文档/分块/事件）是知识库的健康指标，
  应始终可见。

## Agent 设计规范

- **开箱即用**：默认 agent 系统播种、不可删除心智（知识库=全部信源），用户零配置直达对话；
  「创建/绑定」是高级能力，不是使用前提。
- **有据作答**：检索是工具而非硬编码步骤（function-calling 循环，有界步数防跑飞）；
  回答携带 `citations`，空召回给兜底而非幻觉。
- **人格最小化**：persona 只留 `system_prompt / greeting / tools[]`——能靠提示词表达的
  不做成字段，能靠配置中心表达的（top_k/策略）不塞进人格。
- **破坏性操作分层**：常驻界面只暴露可逆动作（侧栏 hover=归档）；彻底删除仅存在于
  归档区——用户在主流程里永远不会误触不可逆操作。
- **会话轻量**：thread 只是消息容器；标题取首问；删除即清，不背记忆包袱。
- Agent 即模型：任何 agent 暴露 OpenAI 兼容端点，站内外行为一致（同一套检索/引用）。

## MCP 作为契约

- **对外**：每个信源即一个 MCP 端点（search/get_entity/get_chunk），HTTP + stdio 双传输；
  连接信息在产品内可复制（不是文档里的秘密）。
- **对内**：agent 对信源检索走进程内暖引擎（快），同一能力经 MCP 对外开放——
  「同源同解」，接口是契约、实现可优化。
- **扩展**：外部 MCP server 挂载即工具（命名空间前缀防撞名），与内置工具在循环中一视同仁；
  单个 server 失联只跳过，不影响对话。

## 文案与语气

- 产品语言：中文、短句、动词开头；不用「您」，不装腔。
- 概念名收敛且全站一致：**对话 / 搜索 / 知识库（信源·文档）/ 设置**——
  同一事物绝不出现第二个叫法。
