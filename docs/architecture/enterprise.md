# 企业级：审计 · 日志 · 权限 · 配置 · 可观测 · 部署

> 目标：**企业可直接部署使用，严谨**。这些不是后补项，而是与功能同权重的一等公民。

## 审计（清晰、不可变）

- 一张 `audit_log`（见 [data-model](data-model.md)），**只增不改不删**。
- 领域服务通过统一钩子写入：`audit.record(actor, action, target, metadata)`。
- 覆盖的关键动作：登录/登出、成员与角色变更、信源/命名空间/灵魂 CRUD、绑定变更、连接器接入与同步、文档删除、**挂载（尤其 full 夺舍）**、写入令牌签发/吊销、问答（可选，含被引用来源）。
- 能力：按 actor/target/时间/动作筛选；导出 CSV/JSON；保留期与归档策略；敏感字段脱敏后入库。
- 原则：审计写入失败**不静默**；关键动作「先审计后执行」或事务内一并提交。

## 日志（结构化、可追踪）

- 结构化日志（JSON 可选），级别可配（现 `core/logging` 已具雏形）。
- 每请求注入 `request_id`；后台任务注入 `job_id`；贯穿日志与响应头，便于全链路排障。
- 任务过程日志落 `job_logs`，前端「处理时间线」可视化。
- 降噪：第三方库（httpx/openai/lancedb）降级到 WARNING（已实现）。
- 不记密钥/原文敏感数据；错误日志带堆栈但响应体只给用户友好信息（已实现错误映射）。

## 权限（RBAC + 多租户隔离）

- 工作空间 = 隔离边界。`memberships.role`：owner/editor/viewer（已具雏形）。
- 资源级：命名空间/灵魂可设可见范围；灵魂绑定决定其可达上下文（天然的最小权限）。
- 平台角色：`users.role` admin/member（首用户 admin，已实现）。
- 令牌：JWT（用户）、ingest 令牌（写入，按信源作用域）、mount 令牌（按灵魂只读作用域）——各自可吊销、限流。

## 配置（清晰、灵活、分层）

优先级：**引擎/平台默认 < 环境变量 < 工作空间设置 < 信源/灵魂覆盖**。
- 现状：`pydantic-settings` + `.env`（`MUSE_*`），信源级 `config.engine` 覆盖 language 等。
- 演进：把「可运行期调整」的项移到 `settings` 表（工作空间级），提供**可视化配置**页（LLM/embedding/检索/连接器），改动**热应用**（重建受影响的 EngineManager 实例），无需重启。
- 密钥：单独的加密存储（KMS/信封加密或至少对 DB 内密钥字段加密）；前端永不下发；审计签发与使用。

## 可观测

- `/system/health`（存活）、`/system/capabilities`（能力探测，已实现）。
- 指标（预留 `/metrics`，Prometheus 风格）：任务队列深度/时延、检索时延、LLM 调用量/失败率、连接器同步成功率。
- 任务面板：queued/running/failed，可重试、可查看 job_logs。
- 就绪探针区分「进程活着」与「依赖（DB/引擎/LLM）可用」。

## 部署（渐进式）

| 档位 | muse 元数据 | 引擎存储 | 适用 |
|---|---|---|---|
| 本地零依赖 | SQLite | LanceDB + SQLite | 个人 / 开发 |
| compose 生产 | Postgres | **pgvector 单库** | 团队 / 中小企业（已提供） |
| 规模化 | Postgres | Elasticsearch / MySQL 拆分 | 大规模检索 |

- compose 已含 web + api + postgres(pgvector)（见 `deploy/`）。
- 演进：Helm/K8s；对象存储放上传件；PG 备份与 PITR；数据驻留/合规（区域化部署）。
- 优雅关闭、健康检查、迁移随启动执行（已具雏形）。

## 安全基线

- 传输 TLS（反代层）；密钥与令牌加密存储、可吊销。
- 连接器出站防 SSRF（私网阻断、超时、大小限）；上传类型与大小校验（已实现上限）。
- 依赖与镜像扫描；最小权限运行（web 容器已用非 root）。
- 数据删除：删除信源可选**级联清除引擎侧向量/图谱**（当前 MVP 保留，企业档提供彻底删除以满足合规）。

> 审计/日志/权限/配置这四件事，决定 muse 是「demo」还是「能进企业的产品」。它们与灵魂、连接器同批推进，而非事后补。
